The highest-impact improvements are these:

Docker deployment is currently broken. In backend/Dockerfile, COPY requirements.txt . looks for a root-level file that does not exist in this repo, and COPY ../automation / COPY ../reports try to copy paths outside the Docker build context, which Docker will reject. On top of that, docker-compose.yml uses a curl healthcheck, but the image never installs curl. That means the “Docker Deployment” path in the README is unlikely to work as written. Files: backend/Dockerfile:9-15, docker-compose.yml:18-22.

The WebSocket runner is unauthenticated, so anyone who can reach /ws/runs/{id}/stream can start a run and stream its output. The REST routes are protected with get_current_user, but the WebSocket handler accepts immediately and then launches _run_and_stream(...) without verifying a token or user first. For a test runner that shells out to pytest, that’s a serious security gap. File: backend/main.py:403-448.

The project ships with hardcoded admin credentials and auto-seeds them on startup. The backend creates admin / qa2024, the README publishes those credentials, and the login screen displays them as a demo hint. That’s fine for a throwaway demo, but it’s not safe for anything called “production-ready,” and it encourages insecure defaults to leak into real deployments. Files: backend/main.py:35-47, README.md:167-172, frontend/index.html:546, automation/conftest.py:17-23.

CORS is misconfigured for browser auth usage. allow_origins=["*"] together with allow_credentials=True is not a valid secure setup for credentialed cross-origin requests, and it’s broader than this app should allow anyway. Even though this frontend currently sends a bearer token manually, this setting is still too permissive and will become a problem as soon as cookies or stricter browser behavior enter the picture. File: backend/main.py:59-65.

The runner updates aggregate counters, but it never persists per-test TestResult rows for real executions. That means /api/runs/{id}/results, flaky analysis, and top-failure analytics are effectively driven by seeded/demo data instead of actual new runs. The data model suggests richer tracking, but run_suite() only increments counts. File: backend/runner.py:96-152.

The frontend presents some analytics as if they are live, but several are hardcoded. The “Top Failures” widget and donut chart use static values instead of API data, so the dashboard can easily mislead users after real runs. File: frontend/index.html:1361-1368, frontend/index.html:1871-1889.
